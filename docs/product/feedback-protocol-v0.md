# Feedback Protocol v0 — YES/NO

## purpose
Define a structured method for collecting, classifying, and using feedback from controlled external YES/NO demo sessions.

## feedback goals
- evaluate product/message clarity
- identify demo friction points
- validate interaction comprehension
- surface repeated concerns without scope drift

## categories of feedback to collect
- product clarity
- interaction clarity
- emotional reaction
- gesture usability
- prototype stability
- perceived differentiation
- confusion points
- trust / credibility concerns

## questions to ask reviewers
- What did you think this product was in the first 30 seconds?
- Was the YES/NO interaction clear?
- Did the gesture model feel intuitive?
- What felt powerful?
- What felt confusing?
- What felt unfinished?
- Would you want to try it again?

## how to record feedback
Capture each session with:
- reviewer type
- date
- demo format
- summary
- quotes
- issues observed
- recommendations

## how to classify feedback
Label each item:
- keep
- investigate
- ignore for now

## how to separate signal from noise
- prioritize repeated patterns across multiple reviewers
- discount isolated preference-only comments
- weight observations tied to demo breakdowns and core clarity

## iteration rules after feedback
- do not react to one-off opinions immediately
- prioritize repeated patterns
- protect core product philosophy
- avoid feature creep from early feedback

## constraints
- conservative interpretation only
- no roadmap inflation from feedback
- no product-direction changes without explicit approval
